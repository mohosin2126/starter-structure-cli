## Template And Preset Conventions

The generated catalog currently uses these output categories:

- `backend-only` (`3`)
- `frontend-only` (`1`)
- `fullstack` (`10`)
- `monorepo-client-server` (`4`)
- `monorepo-turbo-pnpm` (`3`)
- `single` (`10`)

Preset naming is currently one-to-one with generated output:

- preset file: `template-sources/presets/<category>/<slug>.json`
- generated output: `templates/<category>/<slug>`

Current slug conventions:

- stack and tooling tokens are joined with hyphens, for example `react-vite-ts-tailwind`
- TypeScript variants typically use an explicit `-ts` token
- JavaScript variants usually omit `-js` and use the base slug
- database and auth tokens stay in the slug when they materially differentiate a template, for example `express-prisma-mysql-jwt-ts`
- monorepo flavors are encoded at the category level, not inside the slug

Preset composition conventions:

- `base` entries establish the starter foundation
- `layers` overlay language variants, styling, landing/admin variants, or architecture structure
- architecture-heavy templates commonly include shared enterprise layers such as `layers/architecture/fullstack-enterprise-structure/shared`

## Current CLI `--list` Behavior

The current list command:

- discovers templates by walking the generated `templates/` directory
- sorts templates by `category/slug`
- groups output by humanized category label
- prints raw template ids as the selectable reference, for example `fullstack/react-vite-ts-tailwind-express-prisma-mysql`
- does not show descriptions, counts, aliases, or suggested filters in the list output
